Конкретный план проекта v1: агент с метапланированием промптов

1. Тема проекта

Метапланирование промптов как техника построения агентной системы на основе БЯМ.

⸻

2. Исследовательский вопрос

Улучшает ли агент с метапланированием качество решения сложных многошаговых задач по сравнению с baseline-БЯМ?

Этот вопрос фиксируем как основной.

⸻

3. Что считаем успехом

Проект считается успешным, если агент с метапланированием показывает лучший результат, чем baseline, на подходящих задачах.

Подходящие задачи

Подходящими считаем задачи, которые:
	•	сложные;
	•	многошаговые;
	•	требуют выбора стратегии;
	•	выигрывают от планирования, самопроверки и перепланирования.

Что важно понимать

На простых задачах такой подход может быть:
	•	избыточным;
	•	медленным;
	•	неэффективным по затратам.

Это не считается провалом, потому что область применения техники — не простые вопросы, а сложные задачи.

⸻

4. Базовая архитектура агента

Агента делаем без LangGraph, на обычной Python-логике.

Основные роли
	•	Planner
	•	Solver
	•	Critic
	•	при необходимости позже можно добавить Reporter, но на старте он не обязателен.

Planner работает через сочетание шаблонного planner-prompt и пользовательской задачи.
Solver работает на основе пользовательской задачи и плана, построенного Planner.
Critic проверяет решение через critic-prompt с учётом задачи, плана и решения.

⸻

5. Что делает каждый модуль

5.1. Planner

Planner отвечает именно за метапланирование.

Он должен:
	•	определить тип задачи;
	•	понять, подходит ли задача под режим метапланирования;
	•	выбрать стратегию решения;
	•	разбить решение на этапы;
	•	задать критерии качества;
	•	задать условия перепланирования.

Что это значит practically

Planner не решает задачу сам по существу. Он сначала отвечает на вопрос:

Как именно эту задачу лучше решать?

Чем будет реализован Planner

Planner будет вызываться через отдельную функцию, которая обращается к БЯМ с шаблонным planner-prompt’ом.

То есть да, схема именно такая:
	•	есть шаблонный промпт для planner;
	•	функция вызывает БЯМ с этим промптом;
	•	модель возвращает план в структурированном виде.

⸻

5.2. Solver

Solver отвечает за выполнение плана.

Он должен:
	•	взять результат Planner;
	•	пройти по шагам плана;
	•	сформировать черновое решение;
	•	при необходимости разбивать решение по подшагам.

Что важно

Solver не должен подменять Planner.

То есть он не выбирает стратегию заново, а старается следовать плану, который уже был построен.

Solver получает на вход исходную пользовательскую задачу и план, построенный Planner.
На старте Solver можно реализовать как исполнение пользовательского запроса по плану, без отдельного сложного самостоятельного prompt-блока, как у Planner и Critic.

⸻

5.3. Critic

Critic отвечает за проверку качества результата.

Он должен:
	•	проверить, соответствует ли решение задаче;
	•	проверить, выполнен ли план;
	•	проверить полноту;
	•	проверить логическую связность;
	•	решить, нужно ли перепланирование.

Важное понимание

Да: Critic по сути ловит ошибки Planner и Solver.

То есть Critic может заметить, что:
	•	Planner построил слабый или неполный план;
	•	Solver пропустил важный шаг;
	•	итоговое решение противоречиво или неполно.

Чем будет реализован Critic

Точно так же через отдельную функцию, которая вызывает БЯМ с шаблонным critic-prompt’ом.

То есть да, у вас действительно будут отдельные шаблонные промпты как минимум для:
	•	Planner
	•	Critic

Planner и Critic точно используют отдельные шаблонные промпты.
Solver работает на основе пользовательской задачи и плана, построенного Planner.
При необходимости для Solver можно использовать отдельную служебную инструкцию или шаблон вызова, но на старте он не рассматривается как полностью самостоятельный prompt-блок, как Planner и Critic.

⸻

6. Схема решения задачи агентом

Эту схему нужно считать одной из центральных в проекте.

Основная схема

Задача -> Planner -> Solver -> Critic -> при необходимости перепланирование -> финальный ответ

Технически схема вызовов выглядит так:
	•	Planner = planner-prompt + пользовательская задача
	•	Solver = пользовательская задача + план
	•	Critic = critic-prompt + задача + план + решение

В развёрнутом виде
	1.	Пользовательская задача поступает в систему.
	2.	Planner анализирует задачу и строит план.
	3.	Solver решает задачу по плану.
	4.	Critic проверяет результат.
	5.	Если всё хорошо — решение принимается.
	6.	Если обнаружены проблемы — запускается перепланирование.
	7.	После исправления формируется итоговый ответ.

Что это показывает

Эта схема важна, потому что именно она демонстрирует, что у вас не обычный единичный вызов модели, а агент с метапланированием.

⸻

7. Правила перепланирования

Перепланирование запускается, если:
	•	не хватает данных;
	•	Critic находит логическую проблему;
	•	план не покрывает задачу полностью;
	•	ответ получился неполным;
	•	изначально выбранная стратегия оказалась слабой.

Ограничение

Чтобы не получить бесконечный цикл, нужно сразу заложить ограничение:
	•	максимум 1–2 перепланирования на задачу.

⸻

8. Работа с LLM API

Приоритетно используем ChatGPT API, если получится.

Важное решение

Архитектуру делаем так, чтобы замена модели не ломала весь проект.

То есть желательно сделать отдельный модуль доступа к модели, чтобы:
	•	сегодня использовать одну БЯМ;
	•	позже при необходимости заменить её на другую;
	•	при этом не переписывать Planner / Solver / Critic с нуля.

Что важно помнить

Если провайдер БЯМ поменяется, то вероятнее всего придётся переделывать:
	•	модуль доступа к API;
	•	формат вызова модели;
	•	возможно, обработку ответа.

Но логика проекта при этом должна остаться той же.

10. Где и что хранить

10.1. Промпты

Промпты храним как текстовые шаблоны.

Формат
	•	.md или .txt

Почему не JSON

Потому что сами по себе промпты — это просто текст, и в текстовом формате их удобнее читать, редактировать и обсуждать.

Какие промпты понадобятся точно
	•	planner prompt
	•	critic prompt
	•	вероятно solver prompt

⸻

10.2. State и run logs

Что такое state

State — это текущее состояние решения:
	•	задача;
	•	план;
	•	черновик;
	•	замечания critic;
	•	флаг перепланирования;
	•	финальный результат.

Что такое run logs

Run logs — это запись того, как прошёл конкретный запуск агента.

Например:
	•	что получил Planner;
	•	что вернул Solver;
	•	что сказал Critic;
	•	был ли replan.

Формат хранения
	•	.json

Почему JSON нужен здесь

JSON нужен, когда данные структурированы и должны потом читаться программой.

Для человека JSON может быть менее удобен, но для хранения состояния и логов он очень подходит.

⸻

10.3. Датасет

На данный момент датасет планируем делать в табличном формате.

Предварительный формат
	•	.csv

Почему выбираем таблицу

Потому что таблица удобна для:
	•	просмотра набора задач;
	•	сортировки;
	•	фильтрации по категориям;
	•	последующего анализа через pandas;
	•	добавления оценок и результатов сравнения.

Когда бы пригодился JSON

JSON был бы удобнее, если бы у каждой задачи была очень сложная вложенная структура:
	•	много полей;
	•	много дополнительных условий;
	•	вложенные эталоны;
	•	несколько вариантов правильного ответа.

Но на текущем этапе, пока мы делаем понятный рабочий датасет для исследования, таблица выглядит удобнее и проще.

⸻

11. Baseline

Baseline — это одна выбранная заранее сильная БЯМ, с которой мы сравниваем своего агента.

Что важно

Baseline — это не “любая модель вообще”, а конкретный контрольный вариант в эксперименте.

Пока фиксируем так
	•	baseline будет выбран позже отдельно;
	•	это будет одна сильная БЯМ;
	•	сравнение с другими студенческими проектами — это отдельный вопрос и не заменяет baseline.

⸻

12. Оценивание

Пока оценивание оставляем как отложенный блок.

Что уже можно зафиксировать
	•	на текущем этапе главный приоритет — создать работающего агента;
	•	способ сравнения и точные метрики пока не зафиксированы;
	•	ручная оценка выглядит наиболее вероятным вариантом;

13. Итоговая рабочая схема проекта

Что делаем сейчас
	1.	Фиксируем тему и вопрос исследования.
	2.	Фиксируем архитектуру агента.
	3.	Фиксируем стек.
	4.	Делаем Planner / Solver / Critic.
	5.	Настраиваем вызов БЯМ через API.
	6.	Готовим шаблоны промптов.
	7.	Реализуем цикл:
	•	планирование;
	•	решение;
	•	критика;
	•	перепланирование.
	8.	После реализации цикла формируем стартовую тестовую версию датасета.
	9.	Затем датасет расширяется и уточняется.
	10.	После этого отдельно обсуждаем оценивание.
	11.	Потом переходим к эксперименту и статье.
